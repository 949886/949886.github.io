1. @property中assign, copy, retain, strong, weak用法与区别

各个属性应该用在哪？
assign: 基本类型
copy: NSString, NSArray, Block 等
strong: Common object
weak: UIKit / Delegate

strong 与 weak 的区别
strong：为强引用。之所以称之为“强”引用，是因为它会保持住实例，只要强引用还在，实例是不允许被销毁的。
weak：为弱引用。如果一个实例只有弱引用，则其将被释放。为此种属性设置新值时，既不retain也不release，当对象被销毁时属性值被清空（nil out）。

assign, copy, retain的区别
assign: 对基本值类型（scalar type，如NSInteger与C基本类型）进行简单的赋值操作。
copy: 深拷贝新对象，然后释放旧对象。
retain: 浅拷贝赋值对象的地址，其引用计数+1，旧对象引用计数-1。

assign 与 weak 的区别
weak用于对象且在对象被销毁时会被属性值被清空（nil out）。

内存管理

1. Objective-C是如何管理内存的？

一、引用计数

每个对象都会有一个retainCount的引用计数，当引用计数为0时对象就会被销毁。
在MRC下，引用计数需要程序员手动管理。调用retain可使引用计数+1，而release可使引用计数-1。
（MRC下，以alloc init形式的方法创建的对象，其创建后引用计数为1；而例如[NSString string]、[NSArray arrayWithObject:]等工厂方法创建的对象，其创建后引用计数为0，其会在方法结束时被销毁）
在ARC下，内存是由系统管理的，不能直接调用retain或release方法来操作引用计数。编译器会自动生成管理引用计数的代码以及在运行时进行优化。虽然ARC能自动管理内存，但是还是应该注意避免循环引用，以免造成内存泄漏。

二、AutoreleasePool
AutoreleasePool是OC提供的一种内存延迟释放机制，可以通过autorelease方法将对象添加到最近的自动释放池中。当自动释放池释放或者主动调用drain时会向池中的所有对象发送release消息。iOS中，当RunLoop即将进入休眠时(kCFBeforeWaitting)会释放旧的释放池再创建新的释放池，以释放池中的对象。

三、GC(Garbage Collection)机制
类似于Java的垃圾回收机制，系统会自动帮我们回收不使用的对象，但消耗一定的资源。所以，OC的GC只能在Mac上使用，iOS并不支持该功能。

2. AutoreleasePool是什么，底层是如何实现的？

AutoreleasePool是OC提供的一种内存延迟释放机制。

自动释放池以栈的形式实现的，当创建一个新的自动释放池时，它将被添加到栈顶。当一个对象调用autorelease时，系统会将该对象放到栈顶的自动释放池中。当自动释放池释放时，他们就从栈中被移除,并且会给池子里面的所有对象都会做一次release操作。

Runtime

1. 谈谈你对Runtime的理解

一、基本概念

Objective-C是一门动态语言，在OC编译时，会将所有的OC代码编译成运行时C++代码。比如说调用OC方法实际上是调用objc_msgSend方法向该实例发送了一条消息，而Objective-C中的Class在运行时的则是一个指向objc_class结构体的指针，表示一个实例的id其定义则是一个指向objc_object结构体的指针。在objc_object结构体中有一个名为isa的指向objc_class结构体的指针，代表该实例对象运行时的类。objc_class中也有一个指向objc_class的isa指针，指向其meta-class(元类)。meta-class用于储存类本身的属性，如类方法和静态成员变量；另外，objc_class中有一个用于指向父类的super_class指针，以及储存方法表的objc_method_list指针。（其继承关系可以用下图来表示）。

二、动态特性

Objective-C Runtime提供了大量的方法用于对类及其方法进行动态修改，类似于反射。

1.动态类型

即运行时再决定对象的类型。所有的对象都可以由id表示。实际中若一个对象的类型不确定则会用内省(introspection)来检查该对象的类型，常用的内省方法：

  • -(BOOL)isKindOfClass:(Class)class; //是否是class类或class类的子类
  • -(BOOL)isMemberOfClass:(Class)class; //是否是class类(非子类)
  • -(BOOL)conformsToProtocol:(Protocol *)protocol; //是否遵循指定的协议
  • -(BOOL)respondsToSelector:(SEL)selector; //是否能对SEL进行响应

2.消息转发

参考第二题。

2. 谈谈消息转发机制（动态解析）

Objective-C在发送消息时会Runtime会根据实例的isa指针所指向的Class及该Class的父类中找到对应的方法列表，若在最顶层的父类中仍然未找到SEL对应的方法则会触发消息转发机制：

  1. Objective-C会调用该类的 +(BOOL)resolveInstanceMethod:(SEL)sel 和 +(BOOL)resolveClassMethod:(SEL)sel 来指定是否动态添加方法（返回YES则需要在方法内手动通过class_addMethod添加对应方法，返回NO则会进入下一步）。
  2. 若第一步返回NO，则Runtime会检查类是否实现了 -/+(id)forwardingTargetForSelector:(SEL)aSelector 方法(该方法需要返回一个Class，实现-处理转发实例方法，实现+则转发类方法)，用于将该消息转发给其他类处理。
  3. 若第二步返回nil，则Runtime会检查类是否实现了 -(NSMethodSignature *)methodSignatureForSelector:(SEL)aSelector 和 -(void)forwardInvocation:(NSInvocation *)anInvocation方法（若处理的是类方法则为+）。Runtime会先通过methodSignatureForSelector: 来获取方法签名，若此方法返回nil则程序就会抛出unrecognized selector异常，若返回了一个方法签名则会调用forwardInvocation:来手动处理该方法签名对应的invocation。

4.什么是Method Swizzling？

Method Swizzling 可以交换两个方法的实现。方法由两个部分组成，分别是SEL和IMP。SEL为方法的名称，IMP是方法的实现。通过Method Swizzling可以不通过继承而重写原方法，并且还可以保留原方法的实现。

参考：NSHisper - Method Swizzling

5. +load和+initialize方法的用处及区别

+load和+initialize都是用于类初始化的方法。

+(void)load

类载入时会调用。在执行子类的load方法之前，必定会先执行超类的load方法。因为无法判断各个类的载入顺序，故在load方法中使用其他类是不安全的。
如果某个类没有实现load方法，那么无论其超类有没有实现load方法，该类都不会超类的load方法。分类中也可以定义load方法，类的load方法要比其分类中的先调用。
通常Method-Swizzling会在+load方法中进行。

+(void)initialize

类首次被使用之前调用。该方法时「惰性调用」的，也就是说只有当程序用到相关类时才会调用，如果一个类一直都没被使用，那么该方法就一直不会被调用。
如果某个类未实现该方法，而其超类实现了该方法，则该类就会调用超类的initialize方法。鉴于此，一般都会这样实现实现initialize方法：
+(void) initialize { if (self == [ClassName class]) { /* initialize */ } }

Run Loop

1. 谈谈你对Runloop的理解

RunLoop类似于window的消息循环，用来不停地处理输入事件以及调度工作，以及在没有消息处理时休眠以避免资源占用。线程与RunLoop是一一对应的，在RunLoop源码中有一个loopsDic的全局Dictionary，以pthread_t为key，CFRunLoopRef为Value。苹果不允许直接创建RunLoop，只提供了两个方法来自动获取RunLoop：CFRunLoopGetMain() 和 CFRunLoopGetCurrent()。线程刚刚创建的时候并没有RunLoop，若不主动通过这两个方法来获取则不会有RunLoop，而在调用CFRunLoopGetCurrent()后，系统会自动生成一个与当前线程关联的RunLoop并存储在loopsDic中，在线程被销毁时该RunLoop也会随之销毁。在iOS程序启动时会默认启动一个与主线程关联的RunLoop，并初始化loopsDic。

CFRunLoopRef & CFRunLoopModeRef

一个RunLoop包含若干Modes，每个又包含若干个Source、Timer、Observer。mode代表RunLoop的运行模式 ，主要是用来指定运行循环中事件处理的优先级的。而一个RunLoop同时只能指定一个Mode，该Mode被称为CurrentMode。另外，RunLoop还有一个CommonModes属性，RunLoop可以将Mode标记为CommonMode，主线程预置了两个Mode： NSDefaultRunLoopMode和UITrackingRunLoopMode为CommonMode。DefaultMode表示通常状态，TrackingMode表示UIScrollView滑动时的状态。

若创建一个Timer并加入DefaultMode时，则当用户滑动ScrollView时，Runloop会切换到trackingMode，系统为了更好地交互体验就会忽略所有来自该Timer的事件。为了能够让timer在两个Mode钟都被回调，在timer添加到RunLoop时可以将Mode设置为NSRunLoopCommonModes，则timer就会被加入到commonModeItems中，而commonModeItems中的Source/Observer/Timer会被自动加入到被标记为CommonMode的Mode中去，从而使得timer能同时响应两种mode。

参考：深入理解RunLoop

2. AutoreleasePool 什么时候释放？

App启动后系统在主线程RunLoop中注册了两个Observer，其中一个监控的事件是kCFRunLoopEntry(进入RunLoop)，并在回调中创建自动释放池。另一个是监控了两个事件：kCFRunLoopBeforeWaiting(准备进入休眠)时释放旧的释放池并创建新的释放池；kCFRunLoopExit(即将退出RunLoop时)释放自动池。

参考：深入理解RunLoop

设计模式

1. 谈谈你对MCV和MVVM的理解

MVC

MVC是Model-View-Controller(模型-视图-控制器)的缩写，是一种混合设计模式。MVC设计模式改善了程序的耦合性，使得程序更易于复用、扩展和重构。MVC中各个部分的作用为：

  • 模型(Model)：保存数据，定义一些数据的操作及运算。
  • 视图(View)：在程序中所显示的视图、窗口等一些用户可以与之交互并能对用户的操作做出反应的对象。
  • 控制器(Controller)：控制器的主要作用是用来处理业务逻辑，作为View与Model的媒介，负责将Model中的数据传递给View，以及监听View的各种响应事件等。

MVVM

MVVM是Model-View-ViewModel(模型-视图-视图模型)的缩写，安卓中也称为MVP(Model-View-Presenter)，可以说是MVC设计模式的增强版。在很多情况下，如果对MVC使用不当，很可能会让大量的代码都集中在Controller中，造成复用、扩展及维护困难。而在MVVM设计模式中，我们将MVC里Controller中View的逻辑抽出，放到了一个新的View Model中单独处理，使得Model层和View层得以解耦。在MVVM设计模式的使用中，通常会进行双向绑定，即Model变化时，ViewModel会自动更新，而ViewModel变化时，View也会更新。

对于何时使用MCV和MVVM，个人的做法是一般的情况下都采用MVC，而对于一些比较复杂的视图或者一些动态布局的视图会采用MVVM。
