NSThread

NSThread为Objective-C提供了显示的创建和管理线程的API，其是对底层POSIX 线程C语言API的面向对象封装。

线程操作


获取线程

获取主线程： +(NSThread *)mainThread;
获取当前线程： +(NSThread *)currentThread;

创建新线程

-(id)initWithTarget:(id)target selector:(SEL)selector object:(id)argument;
+(void)detachNewThreadSelector:(SEL)aSelector toTarget:(id)aTarget withObject:(id)anArgument;

线程状态

判断当前运行的线程是否为主线程：[NSThread isMainThread];
判断thread是否为主线程：[thread isMainThread];

[thread isExecuting]; //判断thread是否处于运行状态
[thread isFinished]; //判断thread是否执行完毕
[thread isCancelled]; //判断thread是否取消运行

开启&关闭线程

-(void)start; //启动线程，在创建新的线程之后不调用此方法则线程不会运行
-(void)cancel;

另外Objective-C提供了+(void)exit方法终止线程，调用[NSThread exit]可以强制停止当前线程。

挂起线程

挂起线程之后线程将进入阻塞状态。
+(void)sleepUntilDate:(NSDate *)date;
+(void)sleepForTimeInterval:(NSTimeInterval)ti;

NSObject多线程操作

在子线程中执行Selector：
@param aSelector:要执行的方法 arg:SEL的第一个参数或nil

  • -(void)performSelectorInBackground:(SEL)aSelector withObject:(nullable id)arg;

在主线程中执行Selector：
@param aSelector:要执行的方法 arg:SEL的第一个参数或nil modes:RunLoop Modes

  • -(void)performSelectorOnMainThread:(SEL)aSelector withObject:(nullable id)arg waitUntilDone:(BOOL)wait;
  • -(void)performSelectorOnMainThread:(SEL)aSelector withObject:(nullable id)arg waitUntilDone:(BOOL)wait modes:(nullable NSArray *)array;

在指定线程中执行Selector：
@param thr:执行SEL的线程 waitUntilDone:是否在执行完之前一直等待（等待即当前线程会阻塞）

  • -(void)performSelector:(SEL)aSelector onThread:(NSThread *)thr withObject:(nullable id)arg waitUntilDone:(BOOL)wait;
  • -(void)performSelector:(SEL)aSelector onThread:(NSThread *)thr withObject:(nullable id)arg waitUntilDone:(BOOL)wait modes:(nullable NSArray *)array;

在子线程中延时执行Selector：
@param afterDelay 延迟执行的时间（秒）

  • -(void)performSelector:(SEL)aSelector withObject:(nullable id)anArgument afterDelay:(NSTimeInterval)delay;
  • -(void)performSelector:(SEL)aSelector withObject:(nullable id)anArgument afterDelay:(NSTimeInterval)delay inModes:(NSArray *)modes;

递归锁(NSRecursiveLock)

递归锁在Java中也成为可重入锁(ReentrantLock)。顾名思义，即可以在同一线程中进行递归，而不会造成死锁。如果在递归中使用互斥锁则会相互等待释放从而造成死锁。使用方法与NSLock类似，允许在lock与unlock之间进行递归调用。


@synchronized

@synchronized是一种用于创建互斥锁的便捷方式。它能和其他互斥锁做相同的事情，而不需要创建任何锁对象，作为替代，只需要简单地传入一个NSObject对象作为token即可。
传给@synchronized关键字的对象是其唯一标识，若想在多个线程执行@synchronized中的语句可以在不同的线程中向其传递不同的object。在不同的线程中传递相同的object则同时只会有一个线程拿到锁，而其他线程将在该线程执行完毕之前一直阻塞。


NSOperation

Cocoa框架的Operation是一种面向对象方法，用于封装需要处理的异步任务，其是对GCD底层C语言API的面向对象封装。
NSOperation是一个抽象类，无法直接使用。它给了它的子类一个十分有用而且线程安全的方式来建立状态、优先级、依赖性和取消等的模型。Cocoa框架提供了两个继承自NSOperation的子类，NSInvocationOperation 与 NSBlockOperation 以进行异步任务处理。也可以使用自定义的NSOperation的子类以满足更多需求。
注意 若operation中只有单个任务，operation会默认在主线程中执行（NSBlockOperation当有多个任务时，任务会在包括主线程的任意一条线程执行）。实际中operation一般会与OperationQueue一起使用， 不会单独使用Operation执行任务。operation添加到后台OperationQueue中的任务则会在自动在子线程中运行。

NSOperationQueue

Operation Queue类似于Java中的线程池。NSOperation可以被添加到NSOperationQueue中。而NSOperationQueue分为主队列和后台队列，添加到主队列的操作将在主线程中执行，添加到后台队列的操作做将在子线程中执行。

常用属性：

  • NSInteger maxConcurrentOperationCount
    最大并发数。默认为-1代表无限制。 设置为1该队列则会异步串行执行任务。
  • BOOL suspended
    设置该属性为YES即会挂起队列(操作暂停)，再设置为NO即恢复队列(继续执行)。已经开始执行的operation将继续执行，未执行的将被挂起。

常用方法：

  • -(void)addOperation:(NSOperation *)op;
  • -(void)addOperationWithBlock:(void (^)(void))block;
  • -(void)addOperations:(NSArray *)ops waitUntilFinished:(BOOL)wait;

Dispatch Queue

何谓Dispatch Queue？Dispatch Queue可以称作调度队列，在GCD中如果想要处理一个任务，那么只需要将该任务添加到合适的队列中，调度队列就会在合适的线程中自动完成任务。提交到调度队列的任务在系统管理的线程池中执行，而任务在哪个线程中执行并没有明确的保证。
GCD提供下列三种队列：

  • 主队列：在主线程中顺序执行任务。主队列是一种串行队列。
  • 串行队列(Serial Dispatch Queue)：任务一个一个地按先进先出顺序执行。
  • 并发队列(Concurrent Dispatch Queue)：任务按先进先出顺序从队列中取出，但以并行方式执行可以任意顺序完成。

可以通过以下方式创建一个调度队列：
dispatch_get_main_queue()
dispatch_get_global_queue(long identifier, unsigned long flags)
dispatch_queue_create(char * label, dispatch_queue_attr_t attribute)

dispatch_get_main_queue()
功能 返回应用程序主线程的串行队列

主队列会自动被系统创建并关联应用的主线程，所有提交到主队列的任务都会在主线程中执行。与全局并发队列（dispatch_get_global_queue所返回的队列）一样，dispatch_suspend, dispatch_resume, dispatch_set_context不会对其返回的队列有任何效果。

dispatch_get_global_queue(priority, flags)
功能 返回一个系统定义的拥有指定优先级的并发队列。
参数
identifier 表示队列中任务执行的优先级，一般设置为0(Default)即可。该参数的常用枚举值：

  • DISPATCH_QUEUE_PRIORITY_HIGH 高优先级
  • DISPATCH_QUEUE_PRIORITY_DEFAULT 默认优先级
  • DISPATCH_QUEUE_PRIORITY_LOW 低优先级
  • DISPATCH_QUEUE_PRIORITY_BACKGROUND 后台

flags 保留的参数。应始终设置为0。

GCD会在应用中会自动创建4个全局的并发队列（在iOS 5 & OS X 10.7 之前为3个），他们之间只有优先级之间有差异。可以通过dispatch_get_global_queue函数获取该队列。因为这些队列在应用中是全局的，故不需要对其进行retain和release操作，且对其进行的retain及release操作会被自动忽略。

dispatch_queue_create(label, attr)
功能 返回一个可以提交block的队列。
参数
label 一个字符串标签，在Instruments工具、sample、堆栈检测及crash报告中作为其唯一标识符。因为应用程序、库、框架都可以创建其自己的调度队列，推荐使用反向DNS格式(com.example.myqueue)命名。该参数可为NULL。

attr 表示创建并发队列还是串行队列，声明：

  • DISPATCH_QUEUE_SERIAL 或 NULL 创建串行队列
  • DISPATCH_QUEUE_CONCURRENT 创建并发队列

在iOS 6.0之前，GCD并没有使用ARC管理内存，故所有由dispatch_queue_create创建的队列都必须通过调用dispatch_release(dispatch_queue_t)方法手动释放。对应地，若想引用并保持队列则可通过调用dispatch_retain(dispatch_queue_t)增加引用计数。因为现在基本上都是ARC，基本不会使用该方法，并且ARC无法手动调用retain和release方法。

API

dispatch_async

此函数是向调度队列提交任务的最基础的方法，会立即向调度队列提交一个异步执行的block。调用此函数总会在block被提交后立即返回且不等待block被调用。

dispatch_sync

dispatch_sync是一个同步方法，即在任务运行完毕之前当前线程都会被阻塞。

注意 永远不要在相同的queue中调用dispatch_sync方法，这样将会在串行队列中造成死锁。当然，在并发队列中也应该避免。


dispatch_after

dispatch_after一般用于延迟执行一段代码。

通过dispatch_time方法可以创建一个表示延迟的时间dispatch_time_t结构体。该方法第一个参数只能传入DISPATCH_TIME_NOW。第二个参数表示延迟的时间，用NSEC_PER_SEC、NSEC_PER_MSEC、NSEC_PER_USEC分别表示单位秒、毫秒、微妙，2 * NSEC_PER_SEC则表示2秒。

dispatch_suspend / dispatch_resume

通过dispatch_suspend(queue) 和 dispatch_resume(queue)可以挂起和恢复指定的队列。挂起会将未执行的任务暂停，但对队列中已经开始执行的任务没有影响（和NSOperationQueue中的suspended属性一样）。恢复会使挂起的任务继续执行。

dispatch_barrier_async

dispatch_barrier_async(queue, block)函数会保证当前任务在队列中是唯一在执行的一个，即按队列中任务提交的顺序，barrier任务会等前面的任务全部执行完毕后开始执行，执行完毕后队列后的其他任务才会继续执行。可以用来解决读取时的资源竞争问题。


dispatch_semaphore

多个线程同时处理一个资源时，会造成资源竞争，而产生数据不一致的情况。使用dispatch semaphore（semaphore意为信号量）可以有效解决这个问题。虽然通过使用串行队列和dispatch_barrier_async也可以避免这类问题，但dispatch semaphore可以进行更细粒度的控制。
